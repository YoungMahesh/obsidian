OAuth 2.0 with PKCE (Proof Key for Code Exchange) is an extension of the standard OAuth 2.0 [[OAuth 2.0 - Authorization Flow]]  that makes it safe for public clients (like mobile apps, SPAs, or scripts without a secure backend).

---

## Why PKCE exists (the problem it solves)

In a normal OAuth 2.0 flow:

* Your app gets an **authorization code**
* Then exchanges it for an **access token**

👉 Problem:
If someone intercepts that authorization code (e.g., via logs, browser history, or malicious apps), they can exchange it and steal the token.

---

## PKCE adds a “proof” layer

PKCE prevents this by introducing two values:

* **code_verifier** → a secret generated by your client
* **code_challenge** → a hashed version of the verifier (sent early)

Even if an attacker steals the authorization code, they **can’t use it** without the original verifier.

---

## 🔄 Step-by-step flow

### 1. Generate PKCE values (client-side)

* Create a random string → `code_verifier`
* Hash it (usually SHA256 + base64url) → `code_challenge`

```text
code_verifier  →  random_very_secret_string
code_challenge →  BASE64URL(SHA256(code_verifier))
```

`code_verifier` **must be different for every authorization request** (i.e., per login/session)


Script to generate random `code_verifier` and `code_challenge` pair:
```bash
# Step 1: verifier
code_verifier=$(openssl rand -base64 64 | tr -d '=+/' | cut -c1-64)

# Step 2: challenge
code_challenge=$(printf '%s' "$code_verifier" | \
  openssl dgst -sha256 -binary | \
  openssl base64 | \
  tr '+/' '-_' | tr -d '=')

echo "code_verifier: $code_verifier"
echo "code_challenge: $code_challenge"
```


---

### 2. Authorization request

You redirect the user to the authorization server:

```bash
GET /authorize?
  response_type=code
  &client_id=CLIENT_ID
  &redirect_uri=YOUR_REDIRECT
  &code_challenge=XYZ
  &code_challenge_method=S256
```

👉 Important:

* You send **code_challenge**, NOT the verifier

---

### 3. User logs in

* User authenticates on the OAuth server
* Server redirects back with:

```text
?code=AUTHORIZATION_CODE
```

---

### 4. Token exchange (critical PKCE step)

Now your app sends:

```bash
POST /token
grant_type=authorization_code
code=AUTHORIZATION_CODE
code_verifier=original_secret
```

---

### 5. Server verifies

Server does:

```text
SHA256(code_verifier) → compare with stored code_challenge
```

* ✅ If match → issue access token
* ❌ If not → reject

---

## Who generates what?

| Component          | Who generates it | Purpose                  |
| ------------------ | ---------------- | ------------------------ |
| code_verifier      | Your client      | Secret proof             |
| code_challenge     | Your client      | Safe version of verifier |
| authorization code | OAuth server     | Temporary code           |
| access token       | OAuth server     | Final credential         |

---

## Key idea (simple analogy)

Think of it like:

* You send a **lock (code_challenge)** to the server
* Later, you must provide the **key (code_verifier)**

Even if someone steals the lock, they **can’t open it without the key**

---

## Why PKCE is important

Without PKCE:

* Authorization code can be stolen and reused

With PKCE:

* Code is useless without verifier
* Protects against: interception attacks, malicious apps, leaked logs

---

## When should you use PKCE?

👉 Always use PKCE if:

* Mobile apps
* SPA (React, Next.js frontend)
* CLI tools
* Any public client (no secure backend)

👉 Today:

* PKCE is **recommended even for backend apps**
